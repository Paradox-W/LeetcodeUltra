## VS Code Extension Debugging

- Do not use macOS coordinate clicking as the primary way to debug or verify VS Code extensions. It is brittle with Retina scaling, transient welcome dialogs, focus changes, and localized UI.
- Prefer deterministic verification first: unit tests for pure parser/helper logic, `@vscode/test-electron` integration tests for VS Code API behavior, command-line Extension Development Host launches with `--extensionDevelopmentPath`, Debug Adapter Protocol requests, extension-host logs, and CLI/compiler checks.
- Use screenshots as final evidence, not as the main control mechanism. If UI interaction is unavoidable, prefer VS Code commands, extension tests, browser/webview automation with selectors, or direct debug-protocol operations. If manual interaction is required, say so directly.
- For LeetcodeUltra debugger work, verify generated debug files/configuration first, verify CodeLLDB/cppdbg launch behavior second, verify webview/debug-console rendering third, then capture screenshots for proof.
- For Debug Visualizer experiments, verify C++ helper output as JSON first, verify extension parsing/backend behavior second, then use the Extension Development Host and CodeLLDB only for final visual confirmation.

Useful commands:

npm run doctor:debug

- Treat this repository as the authoritative LeetcodeUltra source state.

- Keep every change observable through the owning Git repository before and after editing: use `git -C <repo-root> status --short` and `git -C <repo-root> diff -- <path>` so the change can be reviewed or undone. If Git cannot track the target, stop and explain the blocker instead of editing.

- Keep UI native to VS Code: compact, restrained, tool-like, and dense.
- Avoid duplicated data, decorative borders, or explanatory UI text unless the user asks.
- For visual changes, verify with compile checks and, when possible, a screenshot or extension-host smoke test.
- Do not break Chinese IME. Avoid webview rebuilds or DOM replacement while composition is active.
- Preserve important decisions in this file or a handoff.

## Reuse First

- When a task touches a hard, mature domain such as debuggers, compilers, language runtimes, parsers, protocol clients, formatters, or framework integrations, pause before building or extending a custom implementation.
- First ask whether the user already has a preferred plugin, library, CLI, or workflow for that domain. If the user is not immediately available, inspect the local workspace and installed VS Code extensions, then check official docs or established packages before hand-rolling.

## Architecture Map

The extension is organized around a PureMVC-style notification bus named `BABA`.

High-level flow:

1. `package.json` contributes commands, views, containers, menus, configuration, colors, and activation events.
2. `out/src/extension.js` activates the extension.
3. `extension.js` initializes BABA Mediators and Proxies from `out/src/BABA.js`.
4. VS Code commands and webviews send BABA notifications.
5. Controllers/services handle notifications, call local DAOs, update TreeViews/webviews, or call the child RPC bridge.
6. LeetCode network operations run in a child Node process through `out/src/rpc/childMain.js`.

Primary modules:

- `out/src/extension.js`: activation, command registration, view registration, global timers.
- `out/src/BABA.js`: notification constants, BABA facade wrappers, mediator/proxy base classes.
- `out/src/model/ConstDefind.js`: enums/constants for endpoints, language mappings, tree state, problem state, brick types, dialog options.
- `out/src/model/TreeNodeModel.js`: TreeItem/node model and node-type helpers.
- `out/src/controller/TreeViewController.js`: problem commands such as show, preview, search, pick one, test, submit, favorite, default language.
- `out/src/controller/BricksViewController.js`: repeated-practice tree construction and brick/group actions.
- `out/src/treeData/TreeDataService.js`: `QuestionExplorer` TreeDataProvider and problem list state.
- `out/src/bricksData/BricksDataService.js`: `BricksExplorer` TreeDataProvider and repeated-practice data refresh.
- `out/src/questionData/QuestionDataModule.js`: merges LeetCode problem list, rating data, status, tags, sorting, cache rebuild.
- `out/src/rankScore/RankScoreDataModule.js`: rating data loading.
- `out/src/todayData/TodayDataModule.js`: daily question lookup.
- `out/src/statusBar/StatusBarModule.js`: login status, user contest info, bottom status bar.
- `out/src/statusBarTime/StatusBarTimeModule.js`: per-problem timer.
- `out/src/fileButton/FileButtonModule.js`: CodeLens actions inside generated solution files.
- `out/src/remark/RemarkServiceModule.js`: VS Code Comment API notes and include-template insertion.
- `out/src/debug/*`: local debug support for C++, Python3, and JavaScript.
- `out/src/treeColor/TreeColorModule.js`: file decorations/colors for problem difficulty and brick overdue dates.
- `out/src/workbench/*`: bottom workbench, folding provider, and problem list display options.
- `out/src/companion/CompanionModule.js`: right-side problem/solution/hints/submissions companion webview.
- `out/src/preView/PreviewModule.js`: old preview service, now mostly routes descriptions to Companion.
- `out/src/solution/SolutionModule.js`: old solution/hints service, now mostly routes to Companion.
- `out/src/commitResult/CommitResultModule.js`: receives test/submit output and forwards result payloads to UI/workbench.
- `out/src/childCall/childCallModule.js`: main-process bridge to the RPC child process.
- `out/src/rpc/**`: CLI-like child process implementation and LeetCode network adapters.

## Runtime Data Flow

Problem list flow:

1. `InitEnv` checks Node/RPC setup and switches endpoint.
2. `StartReadData` loads problem data.
3. `QuestionDataModule` calls `ChildCallProxy.get_instance().getAllProblems(...)`.
4. RPC `query -d` resolves through `QueryApi -> chainMgr -> core/cache/retry/leetcode(.cn)`.
5. Rating data is loaded from `resources/data.json` and/or online zerotrac data.
6. `TreeDataService` builds grouped nodes and refreshes `QuestionExplorer`.

Open problem flow:

1. `lcpr.showProblem` or Workbench/Companion command sends `BABACMD_showProblem`.
2. `TreeViewController` resolves node, language, workspace path, and file path.
3. `ChildCallProxy.showProblem(...)` runs RPC `show <qid> -c/-cx -l <lang>`.
4. RPC `ShowApi` fetches problem metadata and renders `resources/templates/codeonly.tpl` or `detailed.tpl`.
5. The generated solution file contains `@lc app=... id=... lang=...`, `@lc code=start/end`, and optional `@lcpr case=start/end` blocks.

Test/submit flow:

1. CodeLens, command palette, tree item, or Workbench sends a test/submit BABA command.
2. `TreeViewController` extracts the file path and optional test case.
3. `ChildCallProxy.testSolution(...)` or `submitSolution(...)` runs RPC `test` or `submit`.
4. `TestApi`/`SubmitApi` normalizes LeetCode results into JSON with `messages` and `system_message`.
5. `CommitResultModule` forwards payloads to result UI and `lcpr.workbench.showResult`.
6. `Workbench` renders status, diagnostics, per-case pass/fail state, and accepted-submit performance charts.

Companion submissions flow:

1. Companion `提交记录` calls `ChildCallProxy.getSubmissionHistory(problemNodeId)`.
2. RPC `submissions <qid>` resolves `SubmissionsApi.fetchList`.
3. `LeetCode.getSubmissions(...)` fetches CN GraphQL `submissionList` or legacy API/page data.
4. Companion normalizes filters and renders rows.
5. Selecting a row calls `submissions <qid> -d <submissionId>`.
6. `SubmissionsApi.fetchDetail` combines `getSubmissionPerformance(...)` and `getSubmission(...)`.
7. Companion renders detail, code, performance charts, and local notes.

AI visualization flow:

1. Workbench only enables visualization after an accepted submit result.
2. `AIVisualizeController` extracts current LeetCode document, code block, line map, and selected test case.
3. Supported patterns, especially sliding-window string problems, prefer deterministic local trace generation.
4. Optional AI binding can improve variable/line mapping.
5. Unsupported patterns fall back to an OpenAI-compatible chat completions endpoint when configured.
6. Trace validation maps AI/local `codeLine` values back to real document lines and highlights them.

## RPC Layer Notes

Main bridge:

- `out/src/childCall/childCallModule.js` owns `ExecuteService`.
- With `useVscodeNode` enabled, the binary path is `out/src/rpc/childMain.js`.
- Otherwise `resources/bin/leetcode` is used; it is only a Node shim requiring `out/src/rpc/childMain`.
- `ExecuteService.callWithMsg(...)` wraps child process execution in VS Code progress notifications.

RPC entry:

- `out/src/rpc/childMain.js` reads `process.argv[2]` as API name.
- `out/src/rpc/factory/apiFactory.js` maps API names: `cache`, `plugin`, `query`, `show`, `star`, `submit`, `submissions`, `test`, `user`.
- `ApiBase.api_argv()` is a small custom argv parser.
- `ReplyUtils` writes JSON or text to stdout for the parent process to parse.

Action chain:

- `chainMgr.init(corePlugin)` loads plugins from `out/src/rpc/actionChain/chainNode`.
- Plugins are sorted by `id` and connected with `setNext(...)`.
- `core` handles filtering, problem lookup, and template export.
- `cache` caches problems, problem detail, rating data, hints, and plugin state.
- `retry` can relogin/retry selected calls if autologin is enabled.
- `leetcode` and `leetcode.cn` handle site-specific network behavior.
- CN and non-CN paths often differ. CN often uses `request`; non-CN often uses `fetch-h2`.

## Persistence And Cache Locations

There are several independent storage layers:

- RPC CLI cache: `~/.lcpr/<endpoint>/cache/*.json`.
- RPC user/session/stat/plugin state: under `~/.lcpr/`.
- Problem solution files: configured by `leetcode-problem-rating.workspaceFolder` and `filePath`.
- Local project data: `<workspace>/.lcpr_data/`.
- Brick data: `<workspace>/.lcpr_data/bricks.json`.
- Brick groups and pick-one tags: `<workspace>/.lcpr_data/group.json`.
- Remark comments: `<workspace>/.lcpr_data/remark/<qid>.json`.
- Submission notes: `<workspace>/.lcpr_data/submission-notes/<problem>.json`, or extension global storage if no workspace folder exists.
- Debug generated files: extension `context.globalStoragePath`.
- Local rating seed: `resources/data.json`.
- Preview-only artifacts: `/Users/paradox/Documents/Leetcode/.codex-preview/`.

Do not assume clearing one storage layer clears the others. `lcpr.deleteCache` and RPC cache deletion only target the RPC cache path, not workspace `.lcpr_data`.

## Known Risk Points

- `out/src/dao/bricksDao.js` has two callbacks like `filter((p) => { p != review_time; })`; these return `undefined` and likely remove every review day instead of only the target.
- `out/src/rpc/actionChain/chainNode/cache.js` references `commUtils.KEYS.hints`, while `commUtils` defines `hints_path`; this may cache hints under an unintended key.
- `package.json` declares VS Code engine `^1.126.0`, while `.vsixmanifest` still declares `^1.57.0`; package metadata may be inconsistent.
- `PreviewModule` and `SolutionModule` still contain old webview rendering paths, but current behavior routes descriptions/solutions/hints into Companion. Do not assume editing old `getWebviewContent()` changes the active UI.
- `node_modules` is third-party dependency boundary. Do not edit it unless debugging an installed dependency patch deliberately.
- `resources/data.json` can be overwritten by rating-data processing scripts or online cache refresh flows.
- Debug flows can modify user solution files by inserting `@lcpr-before-debug` or `@lcpr-after-debug` blocks.
- `hot.sh`, `p.sh`, and `delete.sh` are destructive in this patched installed-extension context. Avoid running them unless the user explicitly asks.

## Source Migration Strategy

If converting this patched installed extension back into a real development project:

1. Clone the upstream MIT-licensed source repository separately.
2. Confirm the upstream version that matches this installed extension as closely as possible, likely around `3.2.4`.
3. Build upstream once without modifications to understand its output shape.
4. Treat this installed directory as the patched behavior reference.
5. Port changes by feature, not by blindly copying `out/src`:
   - package contributions and localization
   - Workbench console
   - test-case editing and IME guards
   - result summary and performance charts
   - AI visualization
   - Companion sidebar
   - submissions RPC API
   - LeetCode submission/performance API enhancements
6. For each feature, update TypeScript source, compile, compare generated output, then manually smoke test in VS Code.
7. Keep commits narrow and named by feature so upstream deltas remain reviewable.
8. Preserve license and attribution before publishing a renamed enhanced extension.

Suggested migration order:

1. Restore upstream build and packaging pipeline.
2. Add package/view/config contributions.
3. Port RPC additions (`submissionsApi`, `apiFactory`, child bridge, LeetCode API methods).
4. Port Workbench result/test-case behavior.
5. Port Companion and route Preview/Solution to it.
6. Port AI visualization.
7. Reconcile localization and README/credits.

Do not use a raw directory copy from this installed extension as the first migration step unless the goal is only a temporary patch. The stronger long-term path is source-level reconstruction with behavior checks against this directory.
